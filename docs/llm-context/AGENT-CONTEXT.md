## Active known issues

- **`text-emphasis` doesn't flip inside `.bg-surface-inverse`.** `--ac-fg-emphasis` is declared at `:root` as `var(--ac-surface-on-primary)` — CSS substitutes the inner var() at the `:root` declaration site, so `.bg-surface-inverse` redeclaring `--ac-surface-on-primary` doesn't propagate to descriptors. `.bg-surface-inverse` already redeclares the numeric `--ac-fg-NN` ramp but NOT the descriptor aliases. Workaround in place: `.text-on-inverse` class (in `brand-color.css`) directly references `var(--ac-surface-on-inverse)`. Real fix is redeclaring `--ac-fg-emphasis` + the other descriptor tokens inside `.bg-surface-inverse` — flagged for a DS hygiene pass.
- **HMR doesn't reliably pick up CSS edits in symlinked workspace packages.** When editing `packages/ds/**` files, restart `pnpm --filter website dev` and `pnpm --filter styleguide dev` to flush the Tailwind v4 + DS cascade rebuild.
- **PayPal Buttons fail silently in local Vite dev.** Smart Buttons render, but `createOrder`'s fetch to `/api/paypal/create-order` 404s because Vite doesn't serve serverless functions. Use `vercel dev` or a Vercel preview deploy for end-to-end testing.
- **Production deploys until env vars are set will break the pay step.** If you push before adding `PAYPAL_*` and `VITE_PAYPAL_CLIENT_ID` to Vercel, the Smart Buttons will render uninitialized and `create-order` will throw 500. Banner the live URL or wait until env is set up before sharing.
- **PayPal currency.** Iceland merchant receives funds in EUR balance; ISK conversion at withdrawal carries a 3–4% FX margin. Set by PayPal account preferences ("Accept all payments without converting").
- **Snipcart's old PayPal NVP/SOAP API permission** still sits in PayPal's legacy UI. PayPal's revocation flow is a maze. With the Snipcart account cancelled, the permission is inert — harmless.
- **Cloudflare BIND import auto-proxies CNAMEs/A records** regardless of the "Proxy imported DNS records" checkbox state. Manual gray-cloud sweep needed post-import. DKIM CNAMEs *must* be unproxied or DKIM signing breaks.
- **`.xyz` deliverability + propagation tax.** New-gTLD scores worse than `.com` in some spam filters (mitigated by full SPF/DKIM/DMARC). Fresh `.xyz` registrations have a noticeable NS-propagation window — Vercel/Proton verifications fail until the `.xyz` registry publishes the Cloudflare nameservers; `dig NS another-creation.xyz +short` is the canary.
- **MailerLite confirmation link blocked by adblockers.** Confirmation emails wrap links through `*.clicks.mlsend.com`, which is on Peter Lowe's blocklist (uBlock Origin, Pi-hole, Brave shields). Subscribers see a warning page with a "Proceed" button — not a hard block but friction. Real fix is paywalled (custom tracking domain) or partially paywalled (disable click tracking). Tracked at backlog 7d.
- **Sanity drafts vs published perspective.** Frontend client uses `perspective: 'published'` — drafts are invisible. Edits saved in Studio go into draft state until you click **Publish** in the top-right pill. "Content edit doesn't show on the site" almost always = doc wasn't published.
- **Modern Sanity Studios require URL registration.** Each public URL (vercel.app fallback, custom domain) registers separately at `sanity.io/manage → project → Studios`. First-load "Connect this studio" prompt is normal; click Register.
- **pnpm v11 strictness** on third-party `postinstall` scripts. Sanity ships esbuild; install fails hard with `ERR_PNPM_IGNORED_BUILDS` until `apps/studio/pnpm-workspace.yaml` has `allowBuilds.esbuild: true`. Fix is already in place — relevant for future Studio re-scaffolding or version bumps.
- **Vercel filesystem routing > rewrites for `/`.** The `build` script in `package.json` renames `dist/index.html` → `dist/app.html` so Vercel's filesystem can't pre-empt the metadata-proxy rewrite for bare `/`. If anyone reverts that change, the homepage will silently serve raw `__TITLE__` placeholders to crawlers. The proxy MUST read `dist/app.html`. Full incident log: `docs/client/kol-client-acyr/01-website/04-metadata/05-incident-crawler-blocked.md`.
- **Vercel Deployment Protection on previews** requires a Protection Bypass token + cookie jar for curl/automation. Token stored in `.env.local` as `VERCEL_AUTOMATION_BYPASS_SECRET`. `scripts/test-meta.sh` auto-applies it for `.vercel.app` URLs.
- **MailerLite "Authorize and Delete conflicts" UI is destructive.** When adding `another-creation.xyz` in MailerLite, the domain-connect flow detects the existing Proton SPF as a "conflict" and offers a one-click "Authorize and Delete." Clicking it WILL wipe Proton's SPF and break outbound mail from `hello@` / `yr@` / etc. The correct fix is to hand-edit the existing SPF TXT record in Cloudflare and merge both includes into one record (DNS only allows ONE `v=spf1` per domain). Pattern: `v=spf1 include:_spf.mlsend.com include:_spf.protonmail.ch ~all`. Same rule applies for any future sender (Brevo, Postmark, etc.) — always merge, never overwrite.

Subscribers using any of those see a block page on the confirm link and never confirm. Fix is a custom tracking domain (backlog 7d). Until that's set up, expect silent drop-off on confirmation from privacy-aware users.
- **`.xyz` first-send Junk landing.** New sender domains on `.xyz` land in Gmail Junk even with full DKIM/SPF/DMARC. Not a config bug — reputation builds with sending history. Inform real subscribers to check Junk and mark-not-junk on first confirmation.

## Critical consistency seams

### Brand display name

`BRAND.name` in `packages/brand-data/config.js` is the single source. Anything user-facing that says "Another Creation" should resolve through `BRAND.name`. If you find a hard-coded string in JSX, that's a bug — fix it on the way past.

### Product `kind` discriminator

`kind: 'pod' | 'handmade'` drives:
- **Routing:** `/shop/:slug` vs `/handmade/:slug` — `ProductDetail.jsx` uses `isPod` to pick `indexHref` and the right buy CTA.
- **Cart eligibility:** POD goes through `addItem`; handmade routes to a `mailto:` enquiry instead.
- **Selectors:** `podProducts()` / `handmadeProducts()` in `shop-data.js`.

If a new `kind` is introduced, all three sites must move together.

### Cart line ID → PayPal order item → Printful sync variant ID

`lineId(slug, size, color)` = `${slug}::${size ?? '-'}::${color ?? '-'}` in `CartContext`. Each Printful product in `printful-products.json` carries `variants[].syncVariantId` keyed by `size + color`. Storage key is `kol.ac.cart.v2` (bumped from v1 when color joined the triple key). The chain at order time:

```
cart line item (slug + size + color)
  → look up syncVariantId by matching size AND color in printful-products.json
  → PayPal order item line (name "Product (Size · Color)" + price + currency)
  → Printful order line (sync_variant_id + quantity)
```

`lookupVariant(slug, size, color)` in `apps/website/api/_lib/products.mjs` is the shared helper used by `validateAndPrice` (create-order), `getShippingRate` (shipping-rates), and the items mapper in capture-order. `color = null` falls back to first-size-match (legacy compatibility for entries without color).

### Server-side price validation

The client's cart can be tampered with (localStorage is editable). `/api/paypal/create-order` MUST recompute the order total from `printful-products.json` and reject if the client-submitted total doesn't match within rounding tolerance. This is the single most important security seam.

### AC DS CSS import order

See ARCHITECTURE §4. Reordering imports in `packages/ds/index.css` silently breaks theming. Any new layer goes at the appropriate cascade position (tokens → brand → utilities → components → framework), not "wherever fits."

### SPF merge discipline

DNS allows ONE `v=spf1` TXT record per domain. Adding a new sender (MailerLite, Brevo, Postmark, etc.) means hand-editing the existing record to append the new `include:`, never deleting + recreating. Current state: `v=spf1 include:_spf.mlsend.com include:_spf.protonmail.ch ~all`. ESP UIs that offer "Authorize and Delete conflicts" buttons are dangerous — they wipe whatever's there. Always go to Cloudflare DNS directly and merge by hand.

### Sanity project ID + dataset name appear in four places

Project ID `ajbrqqhq` and dataset `production` are referenced in:
1. `apps/studio/sanity.config.js` (Studio runtime)
2. `apps/studio/sanity.cli.js` (Studio CLI commands)
3. `apps/website/src/lib/sanity.js` via `VITE_SANITY_PROJECT_ID` / `VITE_SANITY_DATASET` (frontend reads)
4. `scripts/migrate-to-sanity.mjs` via `SANITY_PROJECT_ID` / `SANITY_DATASET` (migration script)

Changing the project ID (e.g., after a Sanity project transfer that re-IDs — does NOT happen on standard transfers; project ID is stable) requires updating all four. The Vercel main-site project also has `VITE_SANITY_PROJECT_ID` / `VITE_SANITY_DATASET` env vars set in all environments — those move with the project too.

## Known gotchas

### Printful auto-fills `external_id` with a hex hash

When you don't set a meaningful `external_id` on a Printful product, Printful fills it with something like `6a07380a565b29`. The sync script's slug heuristic ignores hex-hash external_ids (anything without a hyphen) and slugifies the product name instead. If you want a stable custom URL slug, set `external_id` deliberately in Printful with a hyphen in it (e.g. `acyr-gym-bag`).

### Printful image format

Mockups come down as PNG, not JPG. The sync script detects the extension from the URL and saves accordingly. `public/brand/shop/pod/<slug>.png` for POD products. The hand-authored demos still use `.jpg` — both extensions coexist.

### Vercel SPA rewrite doesn't catch `/api/*`

Vercel's filesystem routing matches BEFORE rewrites. Functions in `api/` are auto-served at `/api/...`, and the catch-all rewrite to `index.html` only fires for paths with no matching file. No `apps/website/vercel.json` change needed for serverless functions.

### Flat shipping constant is duplicated client-side and server-side

`FLAT_SHIPPING_EUR = 10` exists in both `apps/website/api/_lib/products.mjs` (authoritative — used for PayPal order total) and `apps/website/src/pages/site/Checkout.jsx` (for UI summary). If you change one, change the other. When real Printful shipping rates land, both go away — replaced by an async fetch.

**Real-world leak observed in the 2026-05-15 sandbox test:** Printful charges €13.99 to ship the gym bag to Iceland, we charge €10 → €3.99 loss per bag. For non-EU destinations the gap will be larger. Bump the constant or accept the leak until Phase 2.

### Vite dev does not serve `/api/*` routes

`pnpm dev` runs Vite only. PayPal Buttons render but `create-order` will 404. Use `vercel dev` (after `vercel login` + `vercel link`) or push to a Vercel preview branch to test end-to-end locally.

### PayPal sandbox vs live credentials

Two completely separate sets. Don't mix. `PAYPAL_ENV` switches the API base URL (`api-m.sandbox.paypal.com` vs `api-m.paypal.com`) AND the Smart Buttons SDK URL (sandbox client ID won't work against live and vice versa). Test thoroughly in sandbox before flipping.

### Capture idempotency

PayPal's capture endpoint is idempotent on order ID (capturing the same order twice returns the first capture). Printful's order create is **not** idempotent unless you pass `external_id`. Use the PayPal capture ID as the Printful `external_id` to prevent double-fulfillment if the capture endpoint is retried.

### StrictMode double-mount

`apps/website/src/main.jsx` wraps in `<StrictMode>`. The PayPal SDK script must not be loaded twice — use `@paypal/react-paypal-js`'s `PayPalScriptProvider` which handles this correctly, or load the script outside React's lifecycle.
