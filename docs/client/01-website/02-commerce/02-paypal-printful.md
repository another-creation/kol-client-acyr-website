# PayPal + Printful integration — handoff

## Goal

Wire up direct payment + fulfillment for POD merchandise. Pipeline:

```
shop page → add to cart (CartContext) → /checkout (our UI)
  → PayPal Smart Buttons → PayPal approval → /api/paypal/capture-order
  → Printful order created → /checkout/confirmation
```

This replaces the rejected Snipcart approach (see `llm-context/../history.md`).

## Context

- **Merchant location:** Iceland — Stripe unavailable, PayPal Business is the gateway.
- **Currency:** EUR end-to-end. Printful store set to EUR via Preferences. PayPal Business "Accept all payments without converting" enabled — funds settle in EUR balance, ISK conversion happens at withdrawal.
- **Catalog:** static, `src/brand/data/printful-products.json` is generated by `pnpm sync-printful` and committed.
- **Architecture:** SPA + Vercel serverless functions. No database. Order state lives in PayPal + Printful and is fetched on demand.

## What's already done

- Catalog merge: Printful JSON + hand-authored handmade.
- Printful sync script (`scripts/sync-printful.mjs`, `pnpm sync-printful`).
- Cart UI (`/cart`), multi-step checkout UI (`/checkout`), confirmation UI (`/checkout/confirmation`) — all built, all use our design. The card form at the payment step is a placeholder; everything else is real.

### External (still pointing at Snipcart — needs cleanup)

## What needs to happen

### 1. PayPal REST app credentials

1. Log into https://developer.paypal.com with the merchant account.
2. **Sandbox app:** Apps & Credentials → Sandbox → Create App. Note `PAYPAL_CLIENT_ID` + `PAYPAL_SECRET`.
3. **Live app:** Apps & Credentials → Live → Create App. Different client ID + secret. Don't mix them.
4. Sandbox dashboard auto-creates a "personal" buyer account and a "business" merchant account — use those for end-to-end testing.

### 2. Vercel environment variables

Project Settings → Environment Variables. Add:

| Name | Scope | Notes |
|---|---|---|
| `PAYPAL_CLIENT_ID` | Production + Preview | Sandbox ID until ready to flip to live |
| `PAYPAL_SECRET` | Production + Preview | Pair with the same env's client ID |
| `PAYPAL_ENV` | Production + Preview | `sandbox` or `live` |
| `PRINTFUL_TOKEN` | Production + Preview | Same token used locally for the sync script |
| `VITE_PAYPAL_CLIENT_ID` | Production + Preview | **Same value as `PAYPAL_CLIENT_ID`** — Vite exposes this to the client for the Smart Buttons SDK |

The `VITE_` prefix is required for Vite to expose the var to client code. The non-prefixed names stay server-only. **Never** add a `VITE_PAYPAL_SECRET` — it would leak the secret.

### 3. Serverless functions

Add an `api/` directory at the repo root. Vercel auto-routes `api/foo.mjs` to `/api/foo` — no `vercel.json` change needed (the existing catch-all rewrite only fires for non-existent files).

#### `api/paypal/create-order.mjs`

Input (POST body): `{ items: [{ slug, size, qty }], delivery: { country, postcode } }`.

Logic:
1. Load `printful-products.json` from the bundle (`import productsData from '../../src/brand/data/printful-products.json' assert { type: 'json' }` — or read with `fs.readFile`).
2. For each item, look up `printfulProductId` + `syncVariantId` matching the size. Reject if not found.
3. Compute line totals using `printful-products.json` prices (authoritative). Sum.
4. Compute shipping — MVP: flat €20. Phase 2: call Printful shipping rates API with delivery country.
5. POST to PayPal `/v2/checkout/orders` with `intent: CAPTURE`, `purchase_units: [{ amount: { currency_code: 'EUR', value: total, breakdown: { item_total, shipping } }, items: [...] }]`.
6. Return `{ orderID }` to the client.

PayPal auth: OAuth2 client credentials. POST to `/v1/oauth2/token` with the basic-auth header `Basic base64(clientId:secret)` and `grant_type=client_credentials`. Returns an access token good for ~9 hours. Cache it across function invocations within the same warm container (module-level variable with expiry check).

#### `api/paypal/capture-order.mjs`

Input (POST body): `{ orderID, items, delivery, email }`.

Logic:
1. POST to PayPal `/v2/checkout/orders/{orderID}/capture`.
2. On success, extract `purchase_units[0].payments.captures[0].id` — this is the capture ID.
3. POST to Printful `/v2/orders` with:
   - `external_id`: PayPal capture ID (idempotency key)
   - `recipient`: delivery address from request
   - `order_items`: `[{ source: 'catalog', catalog_variant_id: ..., quantity: ... }]` mapped from cart items (or use `source: 'store'` with `sync_variant_id` — verify which Printful v2 prefers).
4. Return `{ captureId, printfulOrderId, status }` to the client.

If the Printful call fails after PayPal capture succeeded, the user has paid but no order was placed. Three options:
- **MVP:** log loudly, return the captureId to the client anyway, manually fulfill from the Printful dashboard.
- **Phase 2:** add a retry-with-backoff and a webhook-driven recovery path.

### 4. PayPal Smart Buttons in `Checkout.jsx`

Install: `pnpm add @paypal/react-paypal-js`.

Wrap the relevant subtree in `<PayPalScriptProvider options={{ "client-id": import.meta.env.VITE_PAYPAL_CLIENT_ID, currency: "EUR", intent: "capture" }}>` — top of `Checkout.jsx` or `main.jsx`.

At the payment step, replace the card form with `<PayPalButtons />`:

```jsx
<PayPalButtons
  style={{ layout: 'vertical', color: 'black', shape: 'rect', label: 'pay' }}
  createOrder={async () => {
    const res = await fetch('/api/paypal/create-order', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ items, delivery }),
    })
    const { orderID } = await res.json()
    return orderID
  }}
  onApprove={async (data) => {
    const res = await fetch('/api/paypal/capture-order', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ orderID: data.orderID, items, delivery, email }),
    })
    const result = await res.json()
    clearCart()
    navigate(`/checkout/confirmation?id=${result.captureId}`, { state: result })
  }}
  onError={(err) => { /* surface error in UI */ }}
/>
```

Strip the existing fake `payment` state, card form, and `PaymentMethodToggle`. The Smart Buttons handle Pay / Pay with Card (and Apple Pay when eligible on the device) as one component.

### 5. `OrderConfirmation.jsx` rewrite

Read from `useLocation().state` (passed by `onApprove`) instead of localStorage. Fallback: if `state` is missing (refresh), refetch by capture ID from a new `GET /api/paypal/order/{id}` endpoint — or just show a polite "thank you" without details.

Remove: `kol.ac.lastOrder` localStorage usage, the fabricated `IS…1234` tracking number.

## Test plan

### Sandbox (development)

1. Deploy with `PAYPAL_ENV=sandbox` and sandbox credentials in Vercel env.
2. Use the sandbox personal account (auto-created in developer dashboard) as the buyer.
3. Add gym bag → checkout → fill address → click PayPal button → log in to sandbox account → approve → bounced back to confirmation.
4. Verify: PayPal sandbox dashboard shows the transaction; Printful dashboard shows the order (the `acyr-test` store can be used for sandbox-Printful too, or use a separate Printful store for sandbox).
5. Refresh the confirmation page — should not re-fulfill (idempotency via capture ID).
6. Test partial failure: temporarily break the Printful token, run a sandbox payment, verify the PayPal capture still succeeds and the error is logged.

### Live

1. Flip `PAYPAL_ENV=live` and swap to live credentials.
2. Set one product to €1 for testing.
3. Run end-to-end with a real PayPal account (different from the merchant account).
4. Verify in PayPal Activity + Printful Orders.
5. Refund from PayPal dashboard (loses the ~€0.35 fixed fee, the percentage is refunded).
6. Reset the product price.

## Constraints / things to know

- **PayPal redirect.** Modern Smart Buttons open PayPal in a popup, not a full-page redirect. The page stays. Some browsers and corporate networks block popups; PayPal falls back to redirect in that case. There is no way to avoid this last hop — it's the bank approval step.
- **EUR balance, ISK at withdrawal.** Iceland merchant receives EUR. Converting EUR → ISK at withdrawal carries a 3–4% FX margin. Avoid frequent small withdrawals; let the balance accumulate.
- **PayPal fees.** International transactions ~3.4% + €0.35 fixed. Domestic-to-Iceland is lower but the customer base is mostly international.
- **Shipping cost realism.** Flat €20 is fine for testing but not for launch — Printful shipping varies from ~€5 (EU bag) to ~€25+ (US jacket). Real rates require the Printful shipping API call from `create-order`.
- **Refund flow** — manual via PayPal dashboard for MVP. Printful order cancellation needs a separate manual step in their dashboard if fulfillment hasn't started yet. Phase 2: build a refund admin flow.

## Phase 2 (not blocking launch)

- `api/paypal/webhook.mjs` — verify signature, handle `PAYMENT.CAPTURE.DENIED`, `CUSTOMER.DISPUTE.CREATED`, `PAYMENT.CAPTURE.REFUNDED`.
- Real shipping rates via Printful shipping API.
- Admin refund flow (small page that calls a `/api/paypal/refund` function).
- Order recovery — if Printful POST fails after capture, retry with backoff; webhook-driven catchup if the function dies.
- Customer accounts + order history — only if a clear product reason emerges.
